# OpenSpec + LLM Wiki 项目知识复利系统 v3.0

> **核心目标**：让每次踩坑、每个决策、每个方案都能沉淀和复用
> **解决什么**：项目 A 踩过的坑，项目 B 不用再踩
> **一句话**：知识不流失，经验可复用，项目越多越快

---

## 一、系统全景

```
┌─────────────────────────────────────────────────────────────┐
│                    知识流转闭环                               │
│                                                             │
│   任务执行 → 产生知识 → 沉淀到 Wiki → 提取通用知识          │
│      ↑                                      ↓              │
│      └──────── 复用已有知识 ← 查询 Wiki ←┘                  │
│                                                             │
│   每循环一次，知识库增长一点                                  │
│   项目越多，复用越多，开发越快                                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

---

## 二、任务生命周期（6 步）

```
PROPOSE → THINK → PLAN → EXECUTE → VERIFY → REVIEW
  提案      思考     规划      执行      验证      复盘
```

### PROPOSE（提案）

定义要做什么。回答 5 个问题：

| # | 问题 | 示例 |
|---|------|------|
| 1 | 一句话问题定义 | "选品 Agent 对模糊需求的解析准确率只有 45%" |
| 2 | 成功标准（可量化） | "准确率从 45% 提升到 80%" |
| 3 | 边界（什么不做） | "不改动现有工作流结构" |
| 4 | 最简单的方案 | "在 Prompt 中加 Few-shot 示例" |
| 5 | 为什么不用最简单的 | "Few-shot 只覆盖常见场景" |

**产出**：`openspec/changes/[task-id]/proposal.md`

---

### THINK（思考）

动手前想清楚。回答 3 个问题 + 追问链：

| # | 问题 | 目的 |
|---|------|------|
| 1 | 可行吗？依赖和风险？ | 避免做到一半发现做不了 |
| 2 | ROI 合理吗？ | 避免花 3 天做价值 1 天的事 |
| 3 | 有现成的吗？ | 避免重复造轮子 |

**追问链**（拿到任何方案后必须追问）：
```
1. 【适用性】适合我的场景吗？
2. 【简化】去掉 50% 还能达到目标吗？
3. 【成本】一半时间能达到多少效果？
4. 【替代】有现成的解决方案吗？
5. 【验证】最小验证成本是多少？
```

**Go/No-Go**：3 个问题都有明确答案且追问链无致命问题 → Go

**产出**：在 `proposal.md` 末尾追加 THINK 结论

---

### PLAN（规划）

拆成可执行任务 + OE 检测。

**任务列表**：

| # | 任务 | 预期产出 | 预计时间 | 依赖 |
|---|------|---------|---------|------|
| 1 | 建 30 个测试用例 | benchmark.json | 0.5 天 | 无 |
| 2 | 跑 baseline | baseline_report.md | 0.5 天 | #1 |
| 3 | 优化 Prompt | prompts.py | 1 天 | #2 |
| 4 | 再测对比 | comparison_report.md | 0.5 天 | #3 |

**OE 检测**（≥3 个未通过 → 方案需简化）：

| # | 检查项 | 状态 |
|---|--------|------|
| 1 | 能简化吗？去掉 50% 还能达到目标？ | ☐ |
| 2 | 最简单的方案是什么？为什么不用？ | ☐ |
| 3 | 有现成的解决方案吗？ | ☐ |
| 4 | 在解决问题，还是在炫技？ | ☐ |
| 5 | 如果只有 1 天，我会怎么做？ | ☐ |
| 6 | 成本值得吗？ | ☐ |

**产出**：`openspec/changes/[task-id]/tasks.md`

---

### EXECUTE（执行）

按任务列表执行，遇到问题记录。

**问题记录格式**：

```markdown
### Problem #1
- **现象**：LLM 输出格式不稳定
- **原因**：Prompt 缺少格式约束
- **解决**：加输出格式约束 + 重试
- **预防**：所有 Prompt 加格式约束
- **适用场景**：所有 LLM 调用场景
```

> **关键**：每个问题都要写"适用场景"，这是后续知识提取和复用的基础。

---

### VERIFY（验证）

对照成功标准，用数据说话。

```markdown
| 成功标准 | 目标 | 实际 | 达成？ |
|---------|------|------|--------|
| 准确率 | 80% | 82% | ✅ |
```

---

### REVIEW（复盘）

5 个问题 + 知识提取。

```markdown
## 复盘

### 1. 一开始对问题的理解对吗？
初始：___ / 实际：___ / 偏差原因：___

### 2. 跳过 Step 0 了吗？
是否跳过：___ / 后果：___

### 3. 有 over-engineering 吗？
过度设计：___ / 最简版本：___

### 4. 用的是什么思想？能迁移吗？
核心思想：___ / 可迁移场景：___

### 5. 重新做会改变什么？
会改变：___ / 不变的核心决策：___
```

## 三、知识沉淀（核心）

### 3.1 知识分层结构

```
wiki/
├── shared/                    ← 通用知识（跨项目复用）
│   ├── patterns/              ← 通用模式（经过验证的方案）
│   ├── errors/                ← 通用错误（踩过的坑）
│   ├── lessons/               ← 通用经验（教训）
│   ├── adr/                   ← 架构决策记录
│   └── index.md               ← 通用知识索引
│
├── projects/                  ← 项目专属知识
│   ├── [项目A]/
│   │   ├── patterns/          ← 项目 A 的方案
│   │   ├── errors/            ← 项目 A 的错误
│   │   └── decisions.md       ← 项目 A 的决策
│   └── [项目B]/
│       └── ...
│
├── pitfalls.md                ← 避坑清单（快速查询）
└── index.md                   ← 总索引
```

### 3.2 知识类型

| 类型 | 存放位置 | 什么时候产生 | 示例 |
|------|---------|------------|------|
| **错误** | wiki/shared/errors/ | EXECUTE 阶段解决问题后 | "LLM 输出格式不稳定 → 加格式约束" |
| **模式** | wiki/shared/patterns/ | VERIFY 阶段验证方案有效后 | "模糊需求处理：澄清 + Few-shot + 验证" |
| **经验** | wiki/shared/lessons/ | REVIEW 阶段复盘后 | "先跑 baseline 再优化，不要盲目改" |
| **决策** | wiki/shared/adr/ | THINK 阶段做技术选型后 | "选 DeepSeek 不选 GPT-4：成本敏感场景" |

### 3.3 知识提取规则

```
什么时候提取：
  - 错误在 2 个以上项目中出现过 → 提取到 shared/errors/
  - 方案在当前项目验证有效 → 提取到 shared/patterns/
  - 经验具有通用性（不依赖特定业务）→ 提取到 shared/lessons/

怎么提取：
  1. 去除项目特定内容（替换项目名、业务术语为通用描述）
  2. 保留技术原理和通用步骤
  3. 添加"适用场景"和"不适用场景"
  4. 添加双向链接（项目页 ↔ 通用页）

什么时候不提取：
  - 只在 1 个项目中出现过 → 留在 projects/ 下
  - 高度依赖特定业务逻辑 → 不提取
  - 还没验证有效 → 不提取
```

## 四、知识复用（价值所在）

### 4.1 复用为什么容易失败

```
3 个根因：

根因 1：记录摩擦 — 记知识太麻烦 → 不记了
根因 2：查找摩擦 — 找知识太麻烦 → 不找了
根因 3：匹配摩擦 — 找到的知识不适用 → 等于没用

解法：让 AI 当知识管理员，把摩擦降到零
```

### 4.2 AI 自动化：3 个检查点

```
┌─────────────────────────────────────────────────────┐
│                                                       │
│  检查点 1：PROPOSE 后 — AI 主动推送                   │
│  ─────────────────────────────────────                │
│  你做：写一句话问题定义                                │
│  AI 做：                                              │
│    1. 语义搜索 wiki/shared/errors/                    │
│    2. 语义搜索 wiki/shared/patterns/                  │
│    3. 语义搜索 wiki/shared/adr/                       │
│    4. 推送 Top 3 相关知识                             │
│  你做：看一眼，标记"用/不用"                          │
│                                                       │
│  检查点 2：REVIEW 后 — AI 自动提取                    │
│  ─────────────────────────────────────                │
│  你做：回答 5 个复盘问题                               │
│  AI 做：                                              │
│    1. 从回答中提取候选知识                            │
│    2. 判断通用性（是否跨项目适用）                    │
│    3. 生成知识草稿（自动填模板）                      │
│    4. 建议提取到 shared/ 或留在 projects/             │
│  你做：确认或修改                                      │
│                                                       │
│  检查点 3：每月 — AI 健康检查                         │
│  ─────────────────────────────────────                │
│  AI 做：                                              │
│    1. 统计知识复用率（被引用次数）                    │
│    2. 标记无人引用的知识（可能过期）                  │
│    3. 检测重复知识（合并建议）                        │
│    4. 生成健康度报告                                  │
│  你做：处理 AI 的建议（删除/合并/更新）              │
│                                                       │
└─────────────────────────────────────────────────────┘
```

### 4.3 匹配精度：靠"适用场景"字段

```
为什么找到的知识经常不适用？
  → 记录时没写"适用场景"，无法判断是否有效

规则：每条知识必须有"适用场景"和"不适用场景"
AI 匹配时对比当前任务特征和知识的适用场景
匹配度高的才推送，低的过滤掉

示例：
  知识："LLM 输出格式不稳定 → 加 JSON Schema 约束"
  适用场景："所有需要结构化输出的 LLM 调用"
  不适用场景："纯对话、创意写作"

  当前任务："优化选品 Agent 的推荐输出格式"
  → 匹配：✅ 适用 → AI 推送
```

### 4.4 复用时机

```
新任务 PROPOSE 阶段：
  → AI 自动搜索并推送相关知识（检查点 1）

新任务 THINK 阶段：
  → 查 wiki/shared/adr/ → 有没有类似的决策参考？

新项目启动时：
  → 查 wiki/projects/[类似项目]/ → 之前的经验是什么？
```

### 4.5 复用效果度量

```markdown
## 本任务的复用情况

| 复用项 | 来源 | 效果 | 节省时间 |
|--------|------|------|---------|
| 模式：模糊需求处理 | wiki/shared/patterns/ | ✅ 直接用 | ~2h |
| 错误：LLM 格式不稳定 | wiki/shared/errors/ | ✅ 预防了 | ~1h |

## 复用统计
- 查阅知识库: X 次
- 成功复用: X 个
- 预防错误: X 个
- 估算节省: X 小时
```

### 4.6 知识健康度

```
每月检查一次（AI 自动生成报告）：

| 指标 | 健康标准 |
|------|---------|
| 新增知识数 | > 0（持续积累） |
| 知识复用率 | > 30%（有复用才有价值） |
| 过期知识数 | = 0（及时清理） |
| 无引用知识数 | < 20%（没人用的知识考虑删除） |
```

### 4.7 最小可行实现

```
不需要复杂系统，用 Claude Code 的 Skill + Hook 即可：

1. Skill：knowledge-search.md
   - 输入：一句话问题定义
   - 动作：语义搜索 wiki/shared/，返回 Top 3
   - 触发：PROPOSE 阶段自动调用

2. Skill：knowledge-extract.md
   - 输入：REVIEW 的 5 个回答
   - 动作：提取候选知识，生成草稿，建议分类
   - 触发：REVIEW 阶段自动调用

3. Hook：每月定时
   - 动作：统计复用率，标记过期/重复知识
   - 触发：手动或定时任务
```

## 五、避坑机制

### 三检查点

```
PROPOSE 后（前置）：
  → 查 pitfalls.md + wiki/shared/errors/
  → "这个任务可能遇到什么已知问题？"
  → 生成避坑清单

EXECUTE 中（过程）：
  → 遇到问题时查知识库
  → "这个问题之前有人解决过吗？"

REVIEW 时（后置）：
  → "这次踩了什么新坑？"
  → 新坑写入 pitfalls.md + wiki/shared/errors/
```

## 八、成功运转保障

> 为什么大多数流程规范最终被废弃？因为它是"额外负担"，不是"工作本身"。
> 以下 4 个条件缺一不可。

### 条件 1：启动成本 < 10 分钟

```
第一个任务启动时，只需要：

1. 建 3 个文件夹（30 秒）
   wiki/shared/errors/
   wiki/shared/patterns/
   wiki/shared/adr/

2. 建 1 个任务文件夹（30 秒）
   openspec/changes/task-001/

3. 写 proposal.md（5 分钟）— 回答 5 个问题
4. 做完写 review.md（5 分钟）— 回答 5 个复盘问题

总计：第一个任务多花 ~10 分钟
第二个任务：~8 分钟（熟练了）
第三个任务：~5 分钟（习惯了）

如果第一个任务就要花 1 小时建 wiki 结构 → 100% 会放弃
```

### 条件 2：每个步骤的产出 = 工作产出

```
❌ 错误做法（额外负担）：
  做完任务 → 额外花 30 分钟写知识文档 → 痛苦 → 放弃

✅ 正确做法（工作即产出）：
  PROPOSE 的 5 个问题 → 就是你做任务前要想清楚的事
  REVIEW 的 5 个复盘 → 就是你做完任务后的自然反思
  问题记录 → 就是你 debug 时本来就要记的

关键洞察：
  不是"做完任务再写文档"
  而是"把思考过程记录下来就是文档"
```

### 条件 3：有即时正反馈

```
第一个任务：
  你写了 proposal → AI 搜索知识库 → 没找到
  → 反馈："知识库为空，你是第一个贡献者！"
  → review 后 AI 提取了 2 条知识 → 存入 wiki
  → 反馈："已沉淀 2 条知识，未来可复用"

第二个任务：
  AI 推送了 1 条相关知识 → 你用了 → 省了 1 小时
  → 反馈："本次复用节省 ~1h"

第三个任务：
  AI 推送了 2 条 → 都用了 → 省了 2 小时
  → 反馈："累计复用节省 ~3h，知识复用率 67%"

第一个任务：你在贡献
第二个任务：你开始受益
第三个任务：你离不开它了
```

### 条件 4：有退化检测

```
信号 1：连续 2 个任务没有走 PROPOSE
  → 流程开始被跳过 → 简化流程或重新明确价值

信号 2：连续 2 个任务 REVIEW 没有提取新知识
  → REVIEW 在走过场 → 检查 REVIEW 质量

信号 3：知识复用率连续 2 个月 < 10%
  → 知识质量差或匹配不准 → 清理过期知识

信号 4：wiki/shared/ 连续 1 个月没有新增
  → 知识积累停滞 → 检查是否在跳过 REVIEW
```

### 一句话

```
体系成功运转 = 启动成本低 + 不额外负担 + 有即时反馈 + 有退化检测

缺任何一个都会失败：
  启动成本高 → 根本不会开始
  额外负担 → 开始了也会放弃
  没有反馈 → 坚持了也感觉不到价值
  没有检测 → 退化了也不知道
```
