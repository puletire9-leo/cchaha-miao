# Claude Code 记忆系统详细分析报告

> 分析日期：2026-04-14

---

## 一、核心设计理念

**核心理念**：只记住无法从代码中推断出来的东西。

### 应该记住 vs 不应记住

| 应该记住 | 不应记住 |
|----------|----------|
| 用户角色、偏好、技能水平 | 代码架构、文件结构 |
| "不要 mock 数据库" | Git 历史、谁改了什么 |
| 项目动态（截止日期、冻结期） | 调试方案（修复已在代码里） |
| 外部系统引用（Linear、Grafana） | 已有 CLAUDE.md 内容 |
| PR 审批偏好 | Ephemeral 任务详情 |

## 二、四种记忆类型详解

### 2.1 User（用户画像）

```
scope: always private
作用: 帮助 Claude 调整协作方式
```

**定义**：记录用户的角色、目标、技能水平和偏好，帮助 Claude 调整协作方式。

**保存时机**：
- 当了解用户的角色、偏好、职责或知识时

**使用方式**：
- 当工作方式应基于用户 profile 或 perspective 时
- 例如：向资深软件工程师解释 vs 向第一次编码的学生解释

**示例**：
```
用户：我是数据科学家，正在调查我们现有的日志系统
Claude 保存：[saves private user memory: user is a data scientist, currently focused on observability/logging]

用户：我写了十年 Go，但这是第一次碰这个仓库的 React 部分
Claude 保存：[saves private user memory: deep Go expertise, new to React and this project's frontend — frame frontend explanations in terms of backend analogues]
```

---

### 2.2 Feedback（行为反馈）

```
scope: default to private
（团队范围仅当是明显的项目级公约，如测试策略、构建不变量）
```

**定义**：用户给出关于如何工作的指导——既包括避免什么，也包括保持什么。

**重要**：不只记录纠正，也记录肯定。如果 Claude 做了一个非显而易见的选择而用户认可了，也会被记住。

**保存时机**：
- 用户纠正方式时（"no not that", "don't", "stop doing X"）
- 用户确认非显而易见的方式工作时（"yes exactly", "perfect, keep doing that"）

**内容结构要求**：
```markdown
## 规则本身

**Why:** 用户给的原因（过去 incident 或偏好）
**How to apply:** 何时/何处应用
```

**示例**：
```
用户：别在回复末尾总结你做了什麼，我能看 diff
Claude 保存：[saves private feedback memory: this user wants terse responses with no trailing summaries. Private because it's a communication preference, not a project convention]

用户：不要在测试中 mock 数据库——上季度 mock 测试通过但生产迁移失败，mock/生产差异掩盖了问题
Claude 保存：[saves team feedback memory: integration tests must hit a real database, not mocks. Reason: prior incident where mock/prod divergence masked a broken migration. Team scope: this is a project testing policy, not a personal preference]

用户：yeah the single bundled PR was the right call here, splitting this one would've just been churn
Claude 保存：[saves private feedback memory: for refactors in this area, user prefers one bundled PR over many small ones. Confirmed after I chose this approach — a validated judgment call, not a correction]
```

---

### 2.3 Project（项目动态）

```
scope: private or team（强烈倾向 team）
作用: 无法从代码/git 推导的项目上下文
```

**定义**：关于项目中正在进行的工作、目标、计划、bugs 或 incidents 的信息。

**关键**：相对日期必须转换为绝对日期���确保记忆不会过时失效。

**保存时机**：
- 了解谁在做什么、为什么、何时截止时
- **重要**：将相对日期转为绝对日期（"周四" → "2026-03-05"）

**内容结构要求**：
```markdown
## 事实或决定

**Why:** 动机（约束、截止日期或 stakeholder 要求）
**How to apply:** 这如何影响建议
```

**示例**：
```
用户：我们周四后冻结所有非关键合并——移动团队要切发布分支
Claude 保存：[saves team project memory: merge freeze begins 2026-03-05 for mobile release cut. Flag any non-critical PR work scheduled after that date]

用户：我们 ripping out 旧 auth middleware 的原因是 legal 说它存储 session token 的方式不符合新的合规要求
Claude 保存：[saves team project memory: auth middleware rewrite is driven by legal/compliance requirements around session token storage, not tech-debt cleanup — scope decisions should favor compliance over ergonomics]
```

---

### 2.4 Reference（外部引用）

```
scope: usually team
作用: 存储外部系统中信息的指针
```

**定义**：指向外部系统中信息的指针，让 Claude 能记住在哪里可以找到最新信息。

**保存时机**：
- 了解外部系统资源及其用途时

**示例**：
```
用户：check Linear project "INGEST" if you want context on these tickets, that's where we track all pipeline bugs
Claude 保存：[saves team reference memory: pipeline bugs are tracked in Linear project "INGEST"]

用户：the Grafana board at grafana.internal/d/api-latency is what oncall watches — if you're touching request handling, that's the thing that'll page someone
Claude 保存：[saves team reference memory: grafana.internal/d/api-latency is the oncall latency dashboard — check it when editing request-path code]
```

## 三、存储结构

### 3.1 目录结构

```
~/.claude/
└── projects/
    └── {项目路径哈希}/
        └── memory/
            ├── MEMORY.md              ← 索引文件（始终加载到上下文）
            ├── user_role.md           ← 用户画像
            ├── feedback_testing.md   ← 行为反馈
            ├── project_*.md             ← 项目动态
            ├── reference_*.md           ← 外部引用
            └── team/                   ← 团队共享记忆（如启用）
                ├── MEMORY.md
                └── ...
```

### 3.2 路径解析顺序

```
1. CLAUDE_COWORK_MEMORY_PATH_OVERRIDE env var（完整路径覆盖）
2. autoMemoryDirectory in settings.json
3. <memoryBase>/projects/<sanitized-git-root>/memory/
   其中 memoryBase = ~/.claude
```

### 3.3 MEMORY.md 格式

```markdown
- [用户角色](user_role.md) — 数据科学家，关注可观测性/日志
- [测试策略](feedback_testing.md) — 集成测试用真实数据库，不 mock
- [合并冻结](project_freeze.md) — 2026-03-05 起冻结非关键合并
- [Bug 追踪](reference_linear.md) — 流水线 bugs 在 Linear INGEST 项目
```

**限制**：最多 200 行或 25KB，超出截断

### 3.4 单个记忆文件格式

```markdown
---
name: {{memory name}}
description: {{单行描述 — 用于决定未来对话中的相关性}}
type: {{user|feedback|project|reference}}
---

{{content}}

**Why:** {{原因}}
**How to apply:** {{应用方式}}
```

## 四、触发保存机制

### 4.1 自动提取（最常用）

```
工作流程：
1. 用户与 Claude 对话
2. Claude 完成回复（无工具调用时）
3. 后台启动「记忆提取子代理」
4. 分析最近的对话内容
5. 识别值得保存的记忆
6. 写入记忆文件 + 更新 MEMORY.md

终端显示：
Memory updated in ~/.claude/projects/.../memory/feedback_testing.md · /memory to edit
```

### 4.2 显式要求

```
用户：记住，这个项目用 bun 不用 npm
→ Claude 立即保存为 feedback 类型

用户：忘记关于合并冻结的记忆
→ Claude 找到并删除相关记忆文件和索引条目

用户：忽略记忆
→ 本次对话中不使用任何记忆内容
```

### 4.3 /memory 命令

```
> /memory
→ 打开文件选择器
→ 用 $EDITOR 或 $VISUAL 编辑
→ 列出所有可编辑的记忆文件
```

### 4.4 /remember 命令

```
> /remember
→ 审查所有自动记忆
→ 提议将合适的条目提升到 CLAUDE.md 或 CLAUDE.local.md
→ 检测重复、过时和冲突的记忆
→ 不会直接修改，所有变更需要用户批准
```

## 五、启用/禁用控制

### 5.1 启用优先级

```
1. CLAUDE_CODE_DISABLE_AUTO_MEMORY
   - 1/true → OFF
   - 0/false → ON

2. CLAUDE_CODE_SIMPLE (--bare) → OFF

3. CCR without persistent storage → OFF
   (no CLAUDE_CODE_REMOTE_MEMORY_DIR)

4. autoMemoryEnabled in settings.json

5. Default: ON
```

### 5.2 禁用方法

| 方法 | 配置 |
|------|------|
| 环境变量 | `CLAUDE_CODE_DISABLE_AUTO_MEMORY=1` |
| 设置文件 | `"autoMemoryEnabled": false` |
| 精简模式 | `--bare` / `CLAUDE_CODE_SIMPLE=1` |

### 5.3 自定义目录

```json
// settings.json
{
  "autoMemoryDirectory": "~/my-claude-memories"
}
```

**注意**：项目级 `.claude/settings.json` **不允许**设置此项（出于安全考虑）

## 六、Fresh度管理

### 6.1 时效处理

| 记忆时效 | 处理方式 |
|----------|----------|
| 今天/昨天 | 直接使用 |
| 超过 1 天 | 陈旧警告，提示验证后引用 |
| 引用文件/函数 | 使用前先 grep 确认存在 |
| 文件路径/函数名的记忆 | 使用前先 grep 确认还存在 |

### 6.2 陈旧检测

- Memory records can become stale over time
- Use memory as context for what was true at a given point in time
- Before answering the user or building assumptions based solely on information in memory records, verify that the memory is still correct and up-to-date
- If a recalled memory conflicts with current information, trust what you observe now — and update or remove the stale memory

## 七、AutoDream 机制

### 7.1 触发条件

当满足以下条件时，Claude 会在后台静默启动"做梦"子智能体进行记忆整合：
- 距上次整合 >= 24 小时
- 期间积累了 >= 5 个会话

### 7.2 四阶段过程

```
1. 定向（Orientation）
2. 收集（Gathering）
3. 整合（Integration）
4. 修剪（Pruning）
```

### 7.3 UI 显示

- 状态栏显示 `dreaming`
- 按 `Shift+Down` 查看进度
- 按 `x` 终止

## 八、引用验证规则

### 8.1 Before recommending from memory

> A memory that names a specific function, file, or flag is a claim that it existed *when the memory was written*. It may have been renamed, removed, or never merged. Before recommending it:

- If the memory names a file path: check the file exists.
- If the memory names a function or flag: grep for it.
- If the user is about to act on your recommendation (not just asking about history), verify first.

### 8.2 关键原则

```
"The memory says X exists" is not the same as "X exists now."

A memory that summarizes repo state (activity logs, architecture snapshots)
is frozen in time. If the user asks about *recent* or *current* state,
prefer git log or reading the code over recalling the snapshot.
```

## 九、不应保存的内容

### 9.1 排除列表

```
- Code patterns, conventions, architecture, file paths, or project structure
  → 这些可以从当前项目状态推导

- Git history, recent changes, or who-changed-what
  → git log / git blame 是权威来源

- Debugging solutions or fix recipes
  → 修复在代码里；commit message ���上���文

- Anything already documented in CLAUDE.md files

- Ephemeral task details: in-progress work, temporary state, current conversation context
```

### 9.2 用户显式要求保存时

如果用户显式要求保存（如 PR list 或 activity summary），询问：
> 有什么 *surprising* 或 *non-obvious* 的？——那就是值得保留的部分。

## 十、访问控制

### 10.1 When to access memories

- 当记忆看起来相关，或用户引用之前对话工作时
- **必须**在用户明确要求检查、回忆或记住时访问
- 如果用户说 *ignore* 或 *not use* memory：按 MEMORY.md 为空处理，不要应用记忆的事实、引用、比较或提及

### 10.2 Proactive 验证

- Memory records can become stale over time
- Use memory as context for what was true at a given point in time
- Before answering the user or building assumptions based solely on information in memory records, verify that the memory is still correct and up-to-date

## 十三、总结

记忆系统是 Claude Code 的核心差异化功能之一，其设计体现了以下原则：

1. **智能记忆** - 只记住无法从代码推导的内容
2. **持久化** - 基于文件的存储，跨会话持久
3. **自动提取** - 无需用户干预，自动学习偏好
4. **Fresh度管理** - 智能检测过时记忆
5. **隐私优先** - 敏感信息默认私有
6. **验证机制** - 引用前验证，避免过时信息误导
